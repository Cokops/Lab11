## Выполненные задания

### Средней сложности

**Задание 1 (5 по методичке): Создать docker-compose.yml, поднимающий Python, Go и Rust сервисы.**
*   Создан файл `MidZad/docker-compose.yml`, который оркестрирует три микросервиса:
    *   **`python_service`**: сервис на Python, создающий файл `from_python.txt`.
    *   **`go_service`**: сервис на Go, создающий файл `from_go.txt`.
    *   **`rust_service`**: сервис на Rust, создающий файл `from_rust.txt`.
*   Все сервисы работают по единому сценарию: создают свой файл и ожидают появление файлов от двух других сервисов.

**Задание 2 (7 по методичке): Использовать volume для обмена данными между контейнерами.**
*   Настроена общая директория для обмена данными:
    *   В `MidZad/docker-compose.yml` объявлен volume `shared_data`.
    *   Все три сервиса монтируют этот volume в директорию `/shared_data`.
    *   Это позволяет им обмениваться файлами (например, `from_python.txt`, `from_go.txt`, `from_rust.txt`) через общую файловую систему.

**Задание 3 (9 по методичке): Ограничить ресурсы (CPU, память) для контейнеров.**
*   Ресурсы для каждого контейнера строго ограничены:
    *   В `MidZad/docker-compose.yml` для каждого сервиса (Python, Go, Rust) установлены лимиты через `deploy.resources.limits`.
    *   **CPU:** `0.5` (половина ядра)
    *   **Память:** `512M` (512 мегабайта)
*   Это обеспечивает предсказуемое поведение и предотвращает один сервис от потребления всех ресурсов.

### Повышенной сложности

**Задание 1 (5 по методичке): Оптимизировать слои Docker-образов для максимального кэширования.**
*   Создан `HardZad/.dockerignore` для исключения из контекста сборки ненужных файлов (`.git`, `__pycache__`, `target/` и т.д.), что уменьшает время передачи контекста и улучшает кэширование.
*   В `HardZad/Dockerfile` реализованы ключевые практики оптимизации слоев:
    *   Копирование `Cargo.toml` и `pyproject.toml` перед копированием исходного кода. Это позволяет Docker кэшировать слой с зависимостями, который не пересобирается при изменении кода.
    *   Кэширование установки `maturin` и `gcc` на этапе `wheel-builder`.
    *   Кэширование установки системных и Python-зависимостей в финальном образе.

**Задание 2 (7 по методичке): Создать multi-stage сборку для Python-приложения с Rust-расширением.**
*   Разработан `HardZad/Dockerfile` с использованием многоэтапной (multi-stage) сборки:
    *   **`rust-builder`**: Первый этап на базе `rust:1.76-slim-bookworm` для компиляции Rust-кода в `cdylib`. Установлены системные зависимости (`pkg-config`, `libssl-dev`, `python3-dev`).
    *   **`wheel-builder`**: Второй этап на базе `python:3.12-slim` для сборки Python wheel-файла с помощью `maturin`. Сюда копируется скомпилированная Rust-библиотека и собирается wheel.
    *   **`runtime`**: Финальный легковесный образ на базе `python:3.12-slim`, куда копируется только созданный wheel-файл и устанавливаются runtime зависимости. Это минимизирует размер финального образа и атакуемую поверхность.

### Структура проекта

- **`MidZad/`** — Мультиязычные сервисы
  - `docker-compose.yml` — Оркестрация Python, Go и Rust сервисов

- **`HardZad/`** — Оптимизированная сборка с Rust-расширением
  - `Dockerfile` — Многоэтапная сборка (multi-stage)
  - `docker-compose.yml` — Оркестрация финального приложения
  - `.dockerignore` — Фильтрация контекста сборки
  - `pyproject.toml` — Конфигурация Python-проекта (PEP 621)
  - `Cargo.toml` — Манифест зависимостей Rust
  - `src/lib.rs` — Код Rust-расширения
  - `src/main.py` — Точка входа Python-приложения
  - `test_functional.py` — Функциональные тесты приложения

- **Корневые файлы**
  - `README.md` — Данная документация
  - `.gitignore` — Игнорируемые файлы
